# 先搜索再实现

当任务可能已有成熟库、框架能力或项目内既有模式时，先搜索再写代码。

## 适用场景

- 需要新增依赖或工具
- 实现通用能力，例如解析、校验、导入导出、图表、文件处理
- 不确定项目里是否已有类似实现
- 需要选择技术方案
- 需要对接第三方 API 或协议

## 工作流程

1. 搜索项目内已有实现、工具函数和约定
2. 搜索官方文档或主流库
3. 比较候选方案的维护状态、体积、许可和复杂度
4. 优先采用已有项目模式
5. 只有在现有方案不合适时才写自定义实现

## 决策标准

| 维度 | 判断 |
|------|------|
| 维护状态 | 近期是否更新、问题讨论是否活跃 |
| 生态适配 | 是否适配当前框架和运行时 |
| 体积 | 是否会明显增加产物体积或冷启动成本 |
| 安全 | 是否有已知漏洞或不安全默认值 |
| 许可 | 是否允许当前项目使用 |
| 复杂度 | 是否比自定义实现更简单 |

## 输出格式

```markdown
## 搜索结论

- 已有项目实现：[文件路径]
- 候选方案：[库/工具]
- 推荐方案：[选择]
- 原因：[简要说明]
- 风险：[维护、体积、安全或迁移风险]
```

## 示例

需求：检查 Markdown 死链。

搜索：npm 中已有 `textlint-rule-no-dead-link`。

结论：优先采用成熟规则，避免手写链接检查器。
