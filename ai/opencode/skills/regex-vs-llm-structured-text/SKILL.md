# 结构化文本解析决策

用于判断应该使用正则、专用解析器，还是 LLM 处理结构化或半结构化文本。

## 默认策略

从确定性方案开始：

1. 已有标准格式：使用标准解析器
2. 规则稳定且格式简单：使用正则或状态机
3. 格式复杂但语法明确：使用 parser 或 AST
4. 格式不稳定、语义判断多：才考虑 LLM

## 选择正则的条件

- 输入格式稳定
- 字段边界明确
- 失败后可以明确报错
- 可用测试覆盖边界情况

```typescript
const match = line.match(/^([^:]+):\s*(.+)$/)
```

## 选择解析器的条件

- 格式有嵌套结构
- 存在转义、引用、注释或多行语义
- 已有成熟库，例如 YAML、CSV、HTML、Markdown、SQL

## 选择 LLM 的条件

- 输入来源不稳定
- 需要语义理解或模糊匹配
- 规则无法穷举
- 可以接受置信度、人工复核或回退路径

## 推荐混合模式

1. 先用正则或解析器处理高置信度样本
2. 对失败或低置信度样本调用 LLM
3. LLM 输出必须再经过 schema 校验
4. 记录失败样本并补充测试

## 风险

- 正则过复杂会变得不可维护
- LLM 输出不稳定，不能直接信任
- 解析失败必须可观测
- 安全敏感输入不能直接拼入 prompt 或命令

## 输出建议

```markdown

原因：[格式稳定性、复杂度、失败处理]
验证：[测试样本和边界条件]
回退：[低置信度或失败时怎么处理]
```
